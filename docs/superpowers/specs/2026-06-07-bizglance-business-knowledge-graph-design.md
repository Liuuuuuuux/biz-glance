# BizGlance 业务知识图谱设计稿

> 目标：把 BizGlance 从“代码结构展示工具”升级为“业务知识图谱工作台”。第一阶段继续依赖 CodeGraph 作为结构底座，LLM 负责语义增强；暂不引入 AST 主解析。

## 1. 项目目标

### 1.1 一句话描述

从本地代码仓库中提取业务对象、业务流转、状态变化和字段血缘，生成可视化的业务知识图谱，帮助开发者和业务人员快速理解系统是“怎么运转的”。

### 1.2 目标用户

- 开发者：快速理解模块边界、接口职责、对象关系和代码证据
- 业务分析人员：看懂业务对象、流程、状态和字段来源
- 交接/审查场景：用图谱代替大量代码翻阅

### 1.3 核心价值

- 默认展示业务语义，而不是代码蜘蛛网
- 每个结论都能回到代码证据
- 允许语义推断，但必须标明置信度
- 先把业务图谱做清楚，再考虑 AST 增强

## 2. 设计原则

- **分层清晰**：结构事实、语义增强、展示层分离
- **证据优先**：所有业务结论必须带证据入口
- **先可用后精确**：先用 CodeGraph + LLM 跑通闭环
- **推断可见**：LLM 推断与确定事实分开展示
- **避免过早重构**：AST 解析作为后续增强，不作为当前主线

## 3. 总体架构

```text
代码仓库
  │
  ▼
CodeGraph 结构提取
  - nodes / edges / codeBlocks / relatedFiles / stats
  │
  ▼
LLM 语义增强
  - 业务对象、业务流、状态变化、字段血缘、置信度
  │
  ▼
BizGlanceDocument
  - businessObjects / flows / statusMutations / fieldLineages / evidences
  │
  ▼
Web 业务图谱展示
  - 域总览 / 对象页 / 流程页 / 状态页 / 血缘页 / 证据侧栏
```

### 3.1 分层职责

**CodeGraph 层**
- 提供文件、符号、调用、引用、路由、代码块和行号
- 不解释业务含义
- 保证输入稳定、可复现、可回溯

**LLM 语义层**
- 从结构事实中提炼业务对象与业务关系
- 识别状态变化、字段来源、业务阶段
- 为每个判断输出解释和置信度

**展示层**
- 把业务对象、流程、状态和证据组织成可读视图
- 让用户先看到业务，再下钻到代码

## 4. 输入设计

### 4.1 CodeGraph 需要产出的字段

建议作为标准输入的字段如下：

- `query`：分析目标说明
- `summary`：CodeGraph 总结
- `nodes[]`：符号节点
- `edges[]`：关系边
- `codeBlocks[]`：关键代码块
- `relatedFiles[]`：相关文件
- `stats`：统计信息

### 4.2 节点建议覆盖

优先保留这些类型：

- `class`
- `method`
- `file`
- `route`
- `controller`
- `service`
- `repository`
- `entity`
- `dto`
- `vo`

### 4.3 边建议覆盖

优先保留这些关系：

- `contains`
- `calls`
- `references`
- `reads`
- `writes`
- `imports`
- `extends`
- `implements`

### 4.4 证据建议

每个关键节点或边最好能回指到：

- `filePath`
- `startLine`
- `endLine`
- `nodeName`
- `route`

## 5. LLM 语义增强设计

### 5.1 要补充的信息

LLM 不负责提取结构事实，只负责补充语义：

- 业务对象中文名
- 所属业务域
- 对象用途描述
- 业务流程阶段
- 对象间关系解释
- 状态变化语义
- 字段血缘说明
- 证据摘要
- 置信度

### 5.2 输出要求

LLM 输出应满足：

- 每个结论都能关联到一个或多个 `evidence`
- `confidence` 必填，推荐值：`high | medium | low`
- 对不确定内容明确标记为推测
- 不要把“代码结构推断”伪装成“业务事实”

### 5.3 推荐的语义补全粒度

- `businessObjects`
  - 技术名、业务名、模块、描述、标签
- `flows`
  - 起点对象、终点对象、关系类型、展示文案
- `statusMutations`
  - 对象、字段、触发点、状态前后变化
- `fieldLineages`
  - 目标字段、源字段、表达式、计算解释

## 6. 现有数据模型演进

### 6.1 当前模型可继续使用

现有 `BizGlanceDocument` 已经可以承载第一阶段结果：

- `businessObjects`
- `flows`
- `statusMutations`
- `fieldLineages`
- `evidences`

### 6.2 建议补强字段

后续可增补这些字段，但不强制第一阶段全部落地：

- `domain`：业务域
- `importance`：重要性
- `entryPoints`：入口点
- `summary`：对象摘要
- `stage`：流程阶段
- `confidenceReason`：置信度原因

## 7. 展示层设计

### 7.1 首页优先展示

首页默认显示“业务流程地图”，而不是代码目录树。

### 7.2 建议的核心视图

**业务域总览**
- 显示各业务域对象数量、关系数量、状态变化数量
- 用于快速建立系统全局认知

**业务对象页**
- 对象定义
- 关联对象
- 相关接口/方法
- 状态变化
- 证据入口

**业务流程页**
- 按流程阶段展示对象如何流转
- 强调“谁触发了什么，结果是什么”

**状态视图**
- 展示对象状态前后变化
- 适合审核、流程系统、单据系统

**字段血缘页**
- 展示字段来源、计算逻辑、落点字段

**证据侧栏**
- 路由、文件、行号、代码片段
- 让业务结论可追溯

### 7.3 业务化展示策略

- 优先使用业务名，技术名作为补充
- 关系文案使用业务语义，例如“生成”“审核”“触发”“归属”
- 默认折叠代码证据，避免首屏信息过载
- 对低置信度结论加标记，不与确定事实混排

## 8. 当前阶段不做的事

- 不做 AST 主解析
- 不做多语言全量覆盖
- 不做增量更新引擎
- 不做复杂自动修复流水线
- 不把 CodeGraph 升级为唯一真相来源

## 9. 分阶段路线

### Phase 1: 业务图谱 MVP

- 继续使用 CodeGraph 作为结构输入
- 使用 LLM 产出业务对象、关系、状态变化、字段血缘
- 输出 BizGlanceDocument
- 完成业务化展示

### Phase 2: 语义增强加强

- 增加更细的领域标签
- 提升证据关联质量
- 完善置信度和解释文本

### Phase 3: AST 增强

- 仅在 CodeGraph 结构不足时补 AST
- 优先用于字段级精度、调用链补全、增量更新

## 10. 验收标准

以下条件满足即可认为第一阶段成立：

- 能从仓库生成业务对象和业务关系图谱
- 首页展示的是业务流程，而不是代码树
- 每个对象和关系都能找到证据
- LLM 推断内容有明显置信度标识
- 不引入 AST 也能跑通完整闭环

## 11. 结论

BizGlance 当前最适合的路线是：

**继续用 CodeGraph 作为结构底座，LLM 做业务语义增强，展示层先做业务知识图谱。**

AST 解析不是当前主线，而是后续在精度和增量能力不足时再补的增强层。
